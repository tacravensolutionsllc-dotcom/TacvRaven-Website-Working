Analytical Methodology

How we produce weekly threat reports

Report Archive Methodology
Overview

Assessment Philosophy

TacRaven's weekly threat reports are designed to provide actionable, defensible assessments using structured analytical techniques. Our goal is to move beyond simple aggregation toward genuine analysis that helps defenders prioritize resources and make informed decisions.

Key Principle: Every assessment includes explicit confidence levels, documented assumptions, and transparent methodology. We distinguish clearly between observed facts and analytical judgments.

Reports are generated automatically every Sunday evening (US Eastern) and published Monday morning, giving security teams a fresh intelligence picture to start their week.

Data Sources

Public Data Sources

We pull from two primary sources, each providing different visibility into the threat landscape:

CISA Known Exploited Vulnerabilities (KEV)

The authoritative catalog of vulnerabilities confirmed to be actively exploited in the wild. CISA adds entries only when they have credible evidence of exploitation, making this our highest-confidence source for exploitation activity.

Update Frequency: Multiple times per week
Data Quality: High — government-verified exploitation evidence
Limitations: May lag behind initial exploitation; focused on US federal relevance

→ CISA KEV Catalog

abuse.ch Feodo Tracker

Tracks command-and-control (C2) infrastructure for major botnet families including Emotet, QakBot, IcedID, and others. Provides IP addresses, ports, malware family attribution, and activity status.

Update Frequency: Near real-time
Data Quality: High — community-verified with researcher attribution
Limitations: Focused on specific malware families; may miss emerging threats

→ Feodo Tracker
Process

Weekly Snapshot Creation

Every Sunday evening, our automated pipeline performs the following steps:

  1. Data Collection: Fresh pulls from both sources with timestamps
  2. Normalization: Converting disparate formats into a unified JSON schema
  3. Deduplication: Removing duplicate indicators across sources
  4. Enrichment: Adding geographic data, malware family context, and MITRE mappings
  5. Analysis: Running structured analytical techniques to generate assessments
  6. Report Generation: Rendering final HTML with all sections

The raw weekly snapshot (JSON) is preserved alongside the HTML report, enabling trend analysis and historical comparisons over time.

Analytical Framework

Structured Analytical Techniques

We employ standardized analytical methods to reduce bias and improve rigor:

Key Assumptions Check (KAC)

Explicitly lists the assumptions underlying each assessment and evaluates their strength based on available evidence. Helps identify where analysis could be wrong.

Analysis of Competing Hypotheses (Mini-ACH)

Tests multiple explanations for observed data by weighing evidence for and against each hypothesis. Prevents premature convergence on a single narrative.

Confidence Level Framework

Applies standardized confidence levels (High/Moderate/Low) with explicit rationale, making analytical uncertainty transparent to the reader.

Standards

Confidence Level Definitions

We use three standardized confidence levels to communicate analytical certainty:

Level Definition Evidence Standard
High Assessment is well-supported by multiple authoritative sources with strong corroboration Multiple independent sources; verified exploitation; clear patterns
Moderate Assessment is reasonably supported but has some gaps or relies on inferential logic Single authoritative source; some corroboration; plausible reasoning
Low Assessment is based on limited data, fragmented sources, or significant assumptions Limited data; weak corroboration; heavy reliance on inference
Framework

MITRE ATT&CK Mapping

We map observed indicators to the MITRE ATT&CK framework to provide tactical context. Our approach prioritizes accuracy over completeness:

Mapping Methodology

  • Curated Malware Mappings: Known malware families (Emotet, QakBot, etc.) are mapped using our curated database of confirmed TTPs
  • KEV Override Mappings: Specific CVEs with documented exploitation patterns have custom technique assignments
  • Default Inference: When no curated mapping exists, we apply conservative defaults (e.g., KEV vulnerabilities → T1190 Exploit Public-Facing Application)
  • Transparency: Inferred mappings are clearly labeled as such in reports

Continuous Improvement: Our mapping files are version-controlled and updated as new intelligence becomes available. Contributors can submit improvements via our GitHub repository.

Caveats

Known Limitations

No analytical product is perfect. Key limitations to consider:

  • Geographic Attribution: IP geolocation reflects hosting infrastructure, not necessarily threat actor origin or target
  • Publication Lag: CISA KEV additions may lag behind initial exploitation by days or weeks
  • C2 Volatility: Botnet infrastructure changes rapidly; status reflects snapshot at collection time
  • Inference Uncertainty: MITRE mappings marked "inferred" use default logic and may not reflect actual adversary TTPs
  • Sector Blindness: Sources don't indicate targeted sectors; all observations are treated as general threats
